Introducao

O Jarvis-WEB-TEAM v2 e uma Skill do OpenCode que implementa um pipeline Spec-Driven Development para projetos web. Ela orquestra subagentes especializados (Execution Manager, Arquiteto, Dev Frontend, Dev Backend, QA, DevOps) para executar especificacoes estruturadas.

Diferenca fundamental: na v2 a skill nao cria especificacao. Ela consome especificacoes prontas (proposal.md, design.md, tasks.md) vindas do OpenSpec ou de qualquer outra ferramenta de planejamento.

Filosofia Spec-Driven

┌──────────────────────────────────────────────────────────┐ │ OpenSpec / Planning Layer │ │ Define: O QUE construir (proposal, design, tasks) │ └────────────────────────┬─────────────────────────────────┘ │ proposal.md ─ design.md ─ tasks.md │ ┌────────────────────────▼─────────────────────────────────┐ │ Jarvis-WEB-TEAM (Execution Layer) │ │ Define: COMO executar cada task │ └──────────────────────────────────────────────────────────┘

Essa separacao permite que a skill funcione com qualquer ferramenta de planejamento: OpenSpec, Claude Code Plan, Cursor Planning, documentos manuais, etc. A skill e um motor de execucao independente.

Pipeline Completo

Fase 1 — Validate Specification
Execution Manager verifica:
  • proposal.md, design.md e tasks.md existem?
  • Os documentos sao consistentes entre si?
  • Todas as tasks possuem Acceptance Criteria?

Se invalido → para e reporta o que falta.

Fase 2 — Build Execution Graph
Execution Manager:
  • Ordena tasks por dependencia
  • Cria execution-state.json
  • Define ordem de execucao
Fase 3 — Architect
Arquiteto:
  • Consome o design.md (nao cria do zero)
  • Detalha decisoes tecnicas (stack, componentes, dados, rotas)
  • Gera estrutura inicial do projeto (pastas, configs)
Fase 4 — Dev Frontend + Dev Backend
Devs:
  • Recebem tasks individuais do Execution Manager
  • Implementam seguindo design.md e Acceptance Criteria
  • Auto-revisao antes de entregar
Fase 5 — Peer Review NOVO
Arquiteto revisa o codigo dos devs:
  • Verifica aderencia ao design.md
  • Verifica qualidade e boas praticas
  • APROVADO → segue para QA
  • REPROVADO → retorna ao dev com relatorio
Fase 6 — QA (Task por Task)
QA valida cada task individualmente:
  • Testa vs Acceptance Criteria
  • Classifica falhas por tipo
  • Loop ate todas passarem

TUDO OK → Fase 7

Fase 7 — DevOps / Delivery
DevOps:
  • Verifica: 100% tasks concluidas e aprovadas pelo QA?
  • Build de producao
  • Relatorio de entrega final
Fase 8 — Aprovacao do Usuario

"O projeto esta de acordo com o que voce esperava?"

  • SIM → finalizado com sucesso
  • NAO → Execution Manager reinicia do passo 3 com novos inputs

Loops de Feedback

Execution Manager │ ┌────────────▼────────────┐ │ Architect │ └────────────│────────────┘ │ ┌────────────▼────────────┐ │ Dev Frontend / Backend │ └────────────│────────────┘ │ ┌────────────▼────────────┐ │ Peer Review │ │ (Arquiteto) │ └────────────│────────────┘ ┌───────┘│ │ │ APROVADO REPROVADO │ │ ┌▼────────────┐ ◄──┘ │ QA │ (volta Dev) └────────│────┘ ┌───┴───┐ │ │ BUG_CODIGO DESIGN_ISSUE │ │ ▼ ▼ Dev Architect │ │ └──┬───┘ │ DevOps │ Aprovacao Usuario
LoopGatilhoOrigemDestino
Loop APeer Review reprovouArquitetoDev
Loop BQA achou BUG_CODIGOQADev
Loop CQA achou DESIGN_ISSUEQAArquiteto
Loop DQA achou REQUISITOQAExecution Manager
Loop EUsuario reprovouUsuarioExecution Manager

Classificacao de Falhas (QA)

O QA classifica cada falha em um dos tres tipos abaixo, determinando para qual agente ela deve ser roteada:

TipoQuando usarDestino
BUG_CODIGO Logica incorreta, estado quebrado, layout quebrado, funcionalidade nao funciona, teste unitario falhou DEV (frontend ou backend)
DESIGN_ISSUE Arquitetura inadequada, componente mal estruturado, decisao tecnica errada, violacao de padrao arquitetural ARQUITETO
REQUISITO Funcionalidade nao atende proposal, AC ambiguo, especificacao incompleta EXECUTION MANAGER

Se o QA encontrar BUG_CODIGO + DESIGN_ISSUE juntos, a prioridade e:

  1. Primeiro DESIGN_ISSUE (Arquiteto corrige a arquitetura)
  2. Depois BUG_CODIGO (Dev corrige o codigo)

Isso evita retrabalho: o dev nao corrige algo que o arquiteto vai mudar.

Ciclo de Vida de uma Task

PENDING ── Execution Manager dispara ──► RUNNING │ Dev concluiu │ ▼ PEER REVIEW ── Arch aprovou ──► QA_TEST │ │ Arch reprovou QA aprovou │ │ ◄──┘ ▼ APPROVED │ QA reprovou │ ▼ FAILED ┌────┴────┐ BUG_CODIGO DESIGN_ISSUE │ │ volta RUNNING volta RUNNING │ Arquiteto revisa

Politica de Retry

Tipo de falhaMaximoAcao ao atingir limite
BUG_CODIGO3 tentativasAborta pipeline
DESIGN_ISSUE3 tentativasAborta pipeline
REQUISITO2 tentativasReporta ao usuario
Build falhou2 tentativasReporta ao Execution Manager
Erro de comunicacaoIlimitadoRetenta automaticamente

Contagem independente: Devs e Arquiteto tem contadores separados. Um DESIGN_ISSUE nao consome as tentativas do Dev.

Exemplo Pratico

Abaixo, um exemplo completo de como usar o OpenSpec + Jarvis-WEB-TEAM para criar um site institucional para uma clinica veterinaria. O exemplo cobre desde a concepcao ate a entrega e manutencao evolutiva.

Cenario: Site para Clinica Veterinaria "Animal Feliz"

Passo 1: Inicializar o OpenSpec no projeto

# Cria a pasta do projeto e inicializa o OpenSpec
mkdir -p ~/projetos/animal-feliz && cd ~/projetos/animal-feliz
openspec init

O comando openspec init cria a estrutura padrao do OpenSpec:

# Estrutura gerada automaticamente
animal-feliz/
├── openspec/
│   ├── specs/              # Fonte da verdade: descreve o sistema atual
│   ├── changes/            # Mudancas propostas (uma pasta por change)
│   │   └── archive/         # Changes concluidas sao movidas para aqui
│   └── config.yaml         # Configuracao do projeto OpenSpec
└── src/                   # Codigo fonte (gerado pela skill posteriormente)
Dica: O openspec init so precisa ser executado uma vez na vida do projeto. Todas as mudancas futuras serao criadas dentro de openspec/changes/.

Passo 2: Propor a primeira mudanca (criacao do site)

# O OpenSpec vai guiar voce na criacao da proposta interativamente
openspec propose "Criar site institucional para Clinica Animal Feliz"

O OpenSpec gera automaticamente todos os artefatos da change:

# Estrutura gerada dentro de openspec/changes/
openspec/changes/site-institucional/
├── proposal.md          # Intencao, escopo, funcionalidades, criterios de aceite
├── specs/               # Delta specs (o que vai mudar no comportamento do sistema)
│   └── site/
│       └── spec.md
├── design.md            # Decisoes tecnicas: stack, arquitetura, etc.
└── tasks.md             # Checklist de implementacao
Importante: O OpenSpec nao substitui a skill. Ele gera a especificacao. A skill executa a especificacao. Sao camadas diferentes e complementares.

Passo 3: Conteudo completo dos artefatos

proposal.md — A intencao do negocio
# Proposta: Site Institucional - Clinica Animal Feliz

## Objetivo
Criar presenca digital profissional para a Clinica Veterinaria Animal Feliz,
permitindo que clientes encontrem informacoes, servicos e entrem em contato.

## Escopo
Incluido:
- Site institucional com 5 paginas (Home, Sobre, Servicos, Equipe, Contato)
- Blog com artigos sobre saude animal
- Formulario de contato com integracao WhatsApp
- SEO basico e responsividade

Excluido:
- Sistema de agendamento online (sera tratado em change futura)
- E-commerce de produtos
- Area do cliente com login

## Funcionalidades
- Pagina inicial com hero e servicos em destaque
- Pagina "Sobre" com historia e missao da clinica
- Pagina "Servicos" com lista de procedimentos e precos
- Pagina "Equipe" com perfil e foto dos veterinarios
- Pagina "Contato" com formulario e mapa
- Blog com listagem e post individual
- Formulario de contato via POST para endpoint
- Botao de WhatsApp flutuante
- SEO (meta tags, sitemap, Open Graph)
- Responsividade (mobile, tablet, desktop)
- Acessibilidade basica (WCAG AA)

## Criterios de Aceite do Projeto
- [ ] Lighthouse Performance > 90
- [ ] Lighthouse Accessibility > 85
- [ ] Lighthouse SEO > 90
- [ ] Responsivo: testado em 3 breakpoints (320px, 768px, 1280px)
- [ ] Formulario de contato envia dados corretamente
- [ ] Botao WhatsApp abre conversa com numero correto
- [ ] Navegacao entre paginas funciona sem erros 404

Cenario Avancado: Abort por limite de tentativas

Se uma mesma task falhar 3 vezes consecutivas, o pipeline e abortado. Exemplo:

[Jarvis-WEB-TEAM] Task T7: Tentativa 1/3 - BUG_CODIGO (numero WhatsApp errado)
[Jarvis-WEB-TEAM] Task T7: Tentativa 2/3 - BUG_CODIGO (validacao do formulario quebrada)
[Jarvis-WEB-TEAM] Task T7: Tentativa 3/3 - BUG_CODIGO (layout quebrado no mobile)

[Jarvis-WEB-TEAM] ❌ PIPELINE ABORTADO
[Jarvis-WEB-TEAM] Task T7 (Contato) atingiu limite de 3 tentativas.
[Jarvis-WEB-TEAM] Relatorio de erro disponivel em: execution-state.json
[Jarvis-WEB-TEAM] Recomendacao: revisar design.md e acceptance criteria da Task T7.
[Jarvis-WEB-TEAM] "Deseja reiniciar a task com ajustes manuais?"
Por que isso e importante? Sem o limite de tentativas, o pipeline poderia entrar em loop infinito gastando tokens e tempo. O aborto forcado obriga uma revisao humana ou ajuste na especificacao, evitando ciclos viciosos.

Comparacao: Como seria no modelo antigo (v1) vs novo (v2)

v1 (Modelo Antigo)

  • PM/PO criava a especificacao do zero em cada execucao
  • Se mudasse um requisito, o pipeline recomeçava inteiro
  • QA testava projeto inteiro de uma vez
  • Feedback do QA era generico: "tem bugs"
  • Sem Peer Review: codigo ia direto para QA
  • Loop infinito de correcao (sem limite de tentativas)

v2 (Modelo Novo)

  • Execution Manager consome especificacao pronta do OpenSpec
  • Mudanca de requisito = nova change (tasks menores)
  • QA testa task por task, regression test no final
  • Feedback classificado: BUG DESIGN REQ
  • Peer Review obrigatorio antes do QA
  • Max 3 tentativas por tipo de agente

Boas Praticas ao usar OpenSpec + Jarvis-WEB-TEAM

Recomendacoes
  1. Escreva Acceptance Criteria claros: Cada task deve ter AC especificos e testaveis. Evite AC vagos como "funciona bem".
  2. Uma change por funcionalidade: Nao acumule multiplas funcionalidades na mesma change. Changes pequenas sao mais faceis de executar, testar e arquivar.
  3. Nao pule o Peer Review: O Peer Review pega problemas de design que o QA nao pegaria (ou pegaria tarde demais). E um investimento que reduz retrabalho.
  4. Classifique as falhas do QA corretamente: Um BUG_CODIGO mal classificado como DESIGN_ISSUE vai para o agente errado e perde tempo.
  5. Arquive changes concluidas: Use openspec archive para mover changes concluidas para o diretorio archive/. Isso mantem o projeto organizado.
