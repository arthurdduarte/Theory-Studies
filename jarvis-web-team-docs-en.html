Introduction

Jarvis-WEB-TEAM v2 is an OpenCode Skill that implements a Spec-Driven Development pipeline for web projects. It orchestrates specialized subagents (Execution Manager, Architect, Dev Frontend, Dev Backend, QA, DevOps) to execute structured specifications.

Key difference: in v2 the skill does not create specs. It consumes ready-made specifications (proposal.md, design.md, tasks.md) from OpenSpec or any other planning tool.

Spec-Driven Philosophy

┌──────────────────────────────────────────────────────────┐ │ OpenSpec / Planning Layer │ │ Defines: WHAT to build (proposal, design, tasks) │ └────────────────────────┬─────────────────────────────────┘ │ proposal.md ─ design.md ─ tasks.md │ ┌────────────────────────▼─────────────────────────────────┐ │ Jarvis-WEB-TEAM (Execution Layer) │ │ Defines: HOW to execute each task │ └──────────────────────────────────────────────────────────┘

This separation allows the skill to work with any planning tool: OpenSpec, Claude Code Plan, Cursor Planning, manual documents, etc. The skill is an independent execution engine.

Complete Pipeline

Phase 1 — Validate Specification
Execution Manager verifies:
  • Do proposal.md, design.md and tasks.md exist?
  • Are the documents consistent with each other?
  • Do all tasks have Acceptance Criteria?

If invalid → stops and reports what is missing.

Phase 2 — Build Execution Graph
Execution Manager:
  • Orders tasks by dependency
  • Creates execution-state.json
  • Defines execution order
Phase 3 — Architect
Architect:
  • Consumes design.md (does not create from scratch)
  • Details technical decisions (stack, components, data, routes)
  • Generates initial project structure (folders, configs)
Phase 4 — Dev Frontend + Dev Backend
Devs:
  • Receive individual tasks from the Execution Manager
  • Implement following design.md and Acceptance Criteria
  • Self-review before delivery
Phase 5 — Peer Review NEW
Architect reviews the devs code:
  • Checks adherence to design.md
  • Checks quality and best practices
  • APPROVED → proceeds to QA
  • REJECTED → returns to dev with report
Phase 6 — QA (Task by Task)
QA validates each task individually:
  • Tests against Acceptance Criteria
  • Classifies failures by type
  • Loop until all pass

ALL OK → Phase 7

Phase 7 — DevOps / Delivery
DevOps:
  • Checks: 100% tasks completed and approved by QA?
  • Production build
  • Final delivery report
Phase 8 — User Approval

"Is the project in accordance with what you expected?"

  • YES → completed successfully
  • NO → Execution Manager restarts from step 3 with new inputs

Failure Classification (QA)

QA classifies each failure into one of three types below, determining which agent it should be routed to:

TypeWhen to useDestination
BUG_CODIGO Incorrect logic, broken state, broken layout, feature not working, unit test failed DEV (frontend or backend)
DESIGN_ISSUE Inadequate architecture, poorly structured component, wrong technical decision, architecture pattern violation ARCHITECT
REQUISITO Feature does not meet proposal, ambiguous AC, incomplete specification EXECUTION MANAGER

If QA finds BUG_CODIGO + DESIGN_ISSUE together, the priority is:

  1. First DESIGN_ISSUE (Architect fixes the architecture)
  2. Then BUG_CODIGO (Dev fixes the code)

This avoids rework: the dev does not fix something the architect will change.

Task Lifecycle

PENDING ── Execution Manager dispatches ──► RUNNING │ Dev completed │ ▼ PEER REVIEW ── Arch approved ──► QA_TEST │ │ Arch rejected QA approved │ │ ◄──┘ ▼ APPROVED │ QA rejected │ ▼ FAILED ┌────┴────┐ BUG_CODIGO DESIGN_ISSUE │ │ returns RUNNING returns RUNNING │ Architect reviews

Retry Policy

Failure TypeMaximumAction when limit reached
BUG_CODIGO3 attemptsAborts pipeline
DESIGN_ISSUE3 attemptsAborts pipeline
REQUISITO2 attemptsReports to user
Build failed2 attemptsReports to Execution Manager
Communication errorUnlimitedRetries automatically

Independent counting: Devs and Architect have separate counters. A DESIGN_ISSUE does not consume the Dev attempts.

Practical Example

Below is a complete example of using OpenSpec + Jarvis-WEB-TEAM to create a corporate website for a veterinary clinic. The example covers from conception to delivery and evolutionary maintenance.

Scenario: Website for "Happy Animal" Veterinary Clinic

Step 1: Initialize OpenSpec in the project

# Creates the project folder and initializes OpenSpec
mkdir -p ~/projects/happy-animal && cd ~/projects/happy-animal
openspec init

The openspec init command creates the standard OpenSpec structure:

# Automatically generated structure
happy-animal/
├── openspec/
│   ├── specs/              # Source of truth: describes the current system
│   ├── changes/            # Proposed changes (one folder per change)
│   │   └── archive/         # Completed changes are moved here
│   └── config.yaml         # Project configuration OpenSpec
└── src/                   # Source code (generated by the skill later)
Tip: openspec init only needs to be run once in the project lifecycle. All future changes will be created inside openspec/changes/.

Step 2: Propose the first change (website creation)

# OpenSpec will guide you through the proposal creation interactively
openspec propose "Create corporate website for Happy Animal Clinic"

OpenSpec automatically generates all change artifacts:

# Structure generated inside openspec/changes/
openspec/changes/corporate-website/
├── proposal.md          # Intent, scope, features, acceptance criteria
├── specs/               # Delta specs (what will change in system behavior)
│   └── site/
│       └── spec.md
├── design.md            # Technical decisions: stack, architecture, etc.
└── tasks.md             # Implementation checklist
Important: OpenSpec does NOT replace the skill. It generates the specification. The skill executes the specification. They are different and complementary layers.

Step 3: Complete artifact content

# Proposal: Corporate Website - Happy Animal Clinic

## Goal
Create professional digital presence for Happy Animal Veterinary Clinic,
allowing clients to find information, services and get in touch.

## Scope
Included:
- Corporate website with 5 pages (Home, About, Services, Team, Contact)
- Blog with pet health articles
- Contact form with WhatsApp integration
- Basic SEO and responsiveness

Excluded:
- Online booking system (will be handled in a future change)
- Product e-commerce
- Customer area with login

## Features
- Home page with hero and featured services
- "About" page with clinic history and mission
- "Services" page with procedure list and prices
- "Team" page with vet profiles and photos
- "Contact" page with form and map
- Blog with listing and individual posts
- Contact form via POST to endpoint
- Floating WhatsApp button
- SEO (meta tags, sitemap, Open Graph)
- Responsiveness (mobile, tablet, desktop)
- Basic accessibility (WCAG AA)

## Project Acceptance Criteria
- [ ] Lighthouse Performance > 90
- [ ] Lighthouse Accessibility > 85
- [ ] Lighthouse SEO > 90
- [ ] Responsive: tested at 3 breakpoints (320px, 768px, 1280px)
- [ ] Contact form sends data correctly
- [ ] WhatsApp button opens conversation with correct number
- [ ] Page navigation works without 404 errors

Advanced Scenario: Abort due to retry limit

If the same task fails 3 consecutive times, the pipeline is aborted. Example:

[Jarvis-WEB-TEAM] Task T7: Attempt 1/3 - BUG_CODIGO (wrong WhatsApp number)
[Jarvis-WEB-TEAM] Task T7: Attempt 2/3 - BUG_CODIGO (form validation broken)
[Jarvis-WEB-TEAM] Task T7: Attempt 3/3 - BUG_CODIGO (broken layout on mobile)

[Jarvis-WEB-TEAM] ❌ PIPELINE ABORTED
[Jarvis-WEB-TEAM] Task T7 (Contact) reached the limit of 3 attempts.
[Jarvis-WEB-TEAM] Error report available at: execution-state.json
[Jarvis-WEB-TEAM] Recommendation: review design.md and acceptance criteria for Task T7.
[Jarvis-WEB-TEAM] "Would you like to restart the task with manual adjustments?"
Why is this important? Without the retry limit, the pipeline could enter an infinite loop wasting tokens and time. The forced abort forces a human review or specification adjustment, preventing vicious cycles.

Comparison: Old model (v1) vs New model (v2)

v1 (Old Model)

  • PM/PO created the spec from scratch on every run
  • If a requirement changed, the pipeline restarted entirely
  • QA tested the entire project at once
  • QA feedback was generic: "has bugs"
  • No Peer Review: code went straight to QA
  • Infinite correction loop (without retry limit)

v2 (New Model)

  • Execution Manager consumes ready spec from OpenSpec
  • Requirement change = new change (smaller tasks)
  • QA tests task by task, regression test at the end
  • Classified feedback: BUG DESIGN REQ
  • Peer Review mandatory before QA
  • Max 3 attempts per agent type

Best Practices when using OpenSpec + Jarvis-WEB-TEAM

Recommendations
  1. Write clear Acceptance Criteria: Each task should have specific, testable AC. Avoid vague AC like "works well".
  2. One change per feature: Do not accumulate multiple features in the same change. Small changes are easier to execute, test and archive.
  3. Do not skip Peer Review: Peer Review catches design problems that QA would not catch (or would catch too late). It is an investment that reduces rework.
  4. Classify QA failures correctly: A misclassified BUG_CODIGO as DESIGN_ISSUE goes to the wrong agent and wastes time.
  5. Archive completed changes: Use openspec archive to move completed changes to the archive/ directory. This keeps the project organized.
