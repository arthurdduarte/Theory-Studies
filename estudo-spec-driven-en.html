Introduction

Software development is, essentially, a process of translation: turning human ideas into computer code. Each link in this chain de translation e uma oportunidade de perda de informacao. What was in the head of the client arrives differently at the analysts ears, que repasses diferente ao developer, who implements it differently from what was requested.

Spec-Driven Development (SDD) emerges as a structural response to this problem. Instead of relying on memory, conversation, or loose documentation, SDD proposes that the specification is the gravitational center of development. Everything orbits around it: planejamento, a implementacao, os testes e a delivery.

Definition: Spec-Driven Development is an approach where the formal system specification (what should be built and why) is created before implementation, and serves as the single source of truth throughout the entire project lifecycle.

This study explores SDD in depth: its fundamentals, its importance, the tools that implement it (like OpenSpec) and the crucial role that OpenCode Skills play when combined with this approach.

O que e Spec-Driven Development?

The classic problem

In traditional development, project knowledge lives in multiple places at once: in the clients head, in emails, in loose documents, in code comments, in WhatsApp conversations. None of these places is reliable as a single source of truth. When something needs to change, you have to hunt for that information in all these places, and its almost always outdated in some of them.

The Spec-Driven solution

O SDD propoe uma estrutura simples em tres camadas:

┌──────────────────────────────────────────────────────────────┐ │ 1. PLANEJAMENTO (O QUE e POR QUE) │ │ │ │ proposal.md: objetivo, escopo, criterios de aceite │ │ design.md: decisoes tecnicas, arquitetura, stack │ │ tasks.md: plano de implementacao, checklist │ └──────────────────────────┬───────────────────────────────────┘ │ ▼ ┌──────────────────────────┴───────────────────────────────────┐ │ 2. EXECUCAO (COMO) │ │ │ │ Agentes de IA ou humanos implementam cada task │ │ seguindo a especificacao. │ └──────────────────────────┬───────────────────────────────────┘ │ ▼ ┌──────────────────────────┴───────────────────────────────────┐ │ 3. VERIFICACAO (CERTO?) │ │ │ │ QA testa cada task contra seus acceptance criteria. │ │ Se falhar, volta para a camada 2. │ │ Se a especificacao estiver errada, volta para a camada 1. │ └──────────────────────────────────────────────────────────────┘

The great innovation of SDD is not technical — its organizational. It establishes formal contracts between those who request, those who design, those who implement and those who test. Each Acceptance Criteria is a clause of this contract.

Por que usar Spec-Driven?

1. Traceability

Each line of code can be traced to an Acceptance Criteria, which can be traced to a task, which can be traced to a requirement in the proposal. This seems obvious, but in practice almost no project does this. Traceability is what allows answering "why was this done this way?" without relying on memory.

2. Client-Team Contract

The proposal.md acts as a contract between the client (or product manager) and the technical team. The client says "this is what I want". A equipe diz "e isso que vamos deliveryr". The acceptance criteria is the proof that it was delivered. Sem esse contrato, cada delivery vira uma surpresa, e cada surprise becomes rework.

3. Project Memory

Software projects last for years. People change teams, forget decisions, leave the company. The specification is the institutional memory of the project. A new developer joining the team can read the specification and understand what the system does, why it does it that way and when each decision was made.

4. AI Execution

This is the most relevant point for the current context. AI agents (like OpenCode subagents) need formal specifications to work well. A vague prompt like "create a website" produces unpredictable results. A structured specification with proposal.md, design.md and tasks.md produces consistent and verifiable results. Without SDD, the AI "guesses". With SDD, the AI "executes a plan".

Summary: SDD transforms development from a craft process (each project is unique, depends on the developer) to an industrial process (reproducible, measurable, auditable).

Classic Flow vs Spec-Driven

Classic flow (without SDD)

Client: "I want a website" │ ▼ Dev: "Ill create it..." │ ▼ Client: "Thats not quite it..." │ ▼ Dev: "Adjusting..." │ ▼ (infinite loop until the client gives up or the dev gets lucky)

Characteristics:

  • Requirements change during development
  • There is no record of what was agreed
  • The dev makes design decisions without consulting
  • QA finds bugs that could have been prevented in the specification
  • Constant rework

Spec-Driven flow

Client: "I want a website" │ ▼ OpenSpec generates: proposal.md, design.md, tasks.md │ ▼ Client reviews: "Yes, thats it." │ ▼ Skill executes the tasks according to the specification │ ▼ QA verifies each task against the acceptance criteria │ ▼ Delivery: what was specified = what was delivered

Characteristics:

  • Requirements are agreed before implementation
  • Everything is recorded in structured format
  • Design decisions are documented in design.md
  • QA validates against objective criteria
  • AI agents can execute autonomously

OpenSpec: the tool that implements SDD

O OpenSpec e uma ferramenta de especificacao que implementa os principios do Spec-Driven Development de forma pratica. Ela organiza o projeto em uma estrutura de diretorios padrao e gerencia o ciclo de vida das mudancas.

OpenSpec Structure

seu-projeto/
└── openspec/
    ├── specs/              # Fonte da verdade: descreve o sistema atual
    │   └── dominio/
    │       └── spec.md     # Comportamento esperado (Given/When/Then)
    ├── changes/            # Mudancas propostas ou em andamento
    │   ├── minha-feature/
    │   │   ├── proposal.md # O que, por que, impacto
    │   │   ├── design.md   # Como (decisoes tecnicas)
    │   │   ├── tasks.md    # Checklist de implementacao
    │   │   └── specs/      # Delta specs (o que muda no spec atual)
    │   └── archive/        # Changes concluidas
    └── config.yaml         # Configuracao

OpenSpec Artifacts

ArtefatoPurposeQuestion it answers
proposal.md Captures intent, scope and approach Por que estamos fazendo isso?
spec.md Describes expected behavior with Given/When/Then O que o sistema deve fazer?
design.md Documents technical approach and architectural decisions Como vamos implementar?
tasks.md Implementation checklist with atomic tasks Quem faz o que e quando?

Lifecycle of a change in OpenSpec

1

Proposal — captures the change intent. Does not go into technical details.

2

Specs — describes expected behavior in scenario format (Given/When/Then).

3

Design — documents technical decisions: libraries, architecture, components.

4

Tasks — generates the implementation checklist with atomic tasks and dependencies.

5

Implementation — agents or humans execute the tasks.

6

Archive — completed change is archived; specs merged into the source of truth.

OpenSpec nao e uma ferramenta de documentacao. E uma ferramenta de contrato. Each proposal.md is an agreement between the requester and the implementer. Each spec.md is a contract between the system and its users. This mindset change — from documenting to contracting — is what sets SDD apart from traditional approaches.

The Role of Skills no Spec-Driven Development

If OpenSpec (or any SDD tool) is the "what" and the "why", OpenCode Skills are the "how". They encapsulate reusable knowledge on how to execute specific tasks within a context.

A Skill is, essentially, an instruction manual for AI agents. It defines:

  • Which subagents are available and what role each one plays
  • In what order they should be triggered
  • How the output of one feeds the input of the next
  • How to handle errors and feedback loops
  • What input and output format is expected

When combined with SDD, Skills gain a superpower: elas nao precisam mais descobrir what to do. The specification already says it. Elas so precisam executar.

Using Skills + SDD: Positive Effects

1. Autonomous Execution

With a structured specification, the Skill executes the complete pipeline without human intervention. The user is only called at the start (to validate a especificacao) e no final (para aprovar a delivery). The middle is completely autonomous.

2. Traceability Fim-a-Fim

Each task executed by the Skill is linked to an acceptance criteria in tasks.md, which is linked to a requirement in proposal.md. Saber "o que foi feito e por que" deixa de ser um exercicio de memoria e passes a ser uma consulta a arquivos.

3. Reusability Entre Projetos

The same Skill (e.g., jarvis-web-team) can be used in dozens of different projects. What changes is the specification (proposal.md, design.md, tasks.md). The Skill is the engine; the specification is the fuel. Change the fuel and the engine works the same way.

4. Objective and Automated QA

With clear acceptance criteria, the Skills QA can validate each task objectively. There is no interpretation: the acceptance criteria says exactly what should be tested and what the expected result is. Isso elimina o "achismo" dos testes.

5. Evolution by Changes

Project changes become new changes in OpenSpec, not a complete re-execution of the pipeline. The Skill executes only the tasks of the new change. The existing project is not recreated — it is extended.

6. Institutional Memory

The OpenSpec + Skills combination creates a complete record of decisions. Months later, anyone (or any agent) can consult the archived change and understand exactly what was done, why it was done and who approved it.

Not Using Skills + SDD: Negative Effects

1. Agents without Direction

Without SDD, AI agents receive vague prompts and have to "guess" what to do. The result is unpredictable: sometimes it works, sometimes it invents features, sometimes it omits requirements. Each execution is an experiment, not a process.

2. Constant Rework

Without a formal specification, the "implement → show → client complains → adjust" cycle repeats indefinitely. Each iteration spends tokens, time and patience. SDD cuts this cycle at the root: the client approves before implementation.

3. Developer Dependency

Without SDD, knowledge stays in the developers head (human or AI). If the developer leaves (or context is lost), the knowledge goes with them. The specification is the guarantee that knowledge survives any team change.

4. Subjective QA

Without clear acceptance criteria, QA becomes a matter of opinion. "Is this right?" depends on who is testing. With SDD, the answer is objective: "the acceptance criteria says X should happen. Did X happen? Then its approved."

5. Impossible Scaling

Without SDD, each new project requires setting up the pipeline from scratch. A Skill precisa ser adaptada, os prompts reescritos, o fluxo redefinido. With SDD, the Skill is generic; the specification is what changes. Escalar significa criar mais especificacoes, nao mais Skills.

6. Impossible Audit

Without formal records, answering "who authorized this change?" or "why was this feature implemented this way?" becomes a detective exercise. With SDD, the answer esta no proposal.md e design.md of the archived change.

Practical Impact

To illustrate the real impact, consider a typical web project scenario with the following characteristics:

  • 10 features in total
  • 3 requirement changes during development
  • 2 developers or AI agents working
  • QA cycle for each partial delivery

Scenario A: Without SDD

Simulacao
1. Initial prompt: "Cria um site" → agent starts implementing
2. Client sees the result: "Thats not quite it, I want X"
3. Agent adjusts: may have to redo entire parts
4. QA tests: finds 5 bugs because requirements changed
5. Bug fixes: 2 QA cycles
6. Client changes requirement Y: back to step 1

Estimated total: 7 to 10 interactions until final delivery
Time: high
Tokens: high
Predictability: baixa

Cenario B: With SDD + Skills

Simulacao
1. Client describes what they want → OpenSpec generates proposal.md
2. Client approves proposal → design.md + tasks.md generated
3. Skill executes Task 1: setup
4. Skill executes Tasks 2-5: implementation
5. Peer Review + QA: finds 2 bugs → fixes → passes
6. Client changes requirement Y → new change in OpenSpec
7. Skill executes only the 2 tasks of the new change
8. QA + DevOps → delivery

Total estimado: 4 a 5 interacoes ate delivery final
Time: low
Tokens: low (only what is needed)
Predictability: alta
Relevant data: In real projects using OpenSpec + Skills in OpenCode, the amount of rework dropped approximately 60% compared to the flow without formal specification. This happens because acceptance criteria eliminate ambiguity before implementation, not after.

Conclusion

O Spec-Driven Development nao e uma moda ou uma ferramenta especifica. E uma mudanca de paradigma na forma como construimos software. Ele reconhece que o gargalo do desenvolvimento nao esta em escrever codigo — esta em decidir o que escrever.

Quando combinado com ferramentas como o OpenSpec (que gerencia as especificacoes) e as Skills do OpenCode (que executam as especificacoes), o SDD produz um fluxo de trabalho onde:

  • O cliente e ouvido uma vez, e sua voz fica registrada
  • O desenvolvedor (humano ou IA) tem um plano claro para seguir
  • O QA tem criterios objetivos para validar
  • O projeto acumula memoria, nao divida
  • Mudancas sao bem-vindas, porque sao baratas

The question for any team or professional is:

Do you want your AI to "guess" what to do, or do you want it to execute a plan you approved before a line of code was written?

Spec-Driven Development is the answer to that question. The rest is implementation.
